
## 1、背景

在应用开发中，表单、表格、详情页、导出、打印等前端/服务端功能，大量重复地围绕数据模型进行增删改查、展示和交互。开发方式存在以下痛点：

1. **重复开发**：同一模型对应的表单、表格、过滤器、导出等需要多次编写相近的逻辑与配置。
2. **变更成本高**：模型字段增减、类型修改、命名调整等变更，会导致上下游（前端展示、后端逻辑、报表导出等）多处改动，极易遗漏或不一致。
3. **跨库/跨模块协作困难**：业务模型可能分布在单库、多库甚至不同模块，缺乏统一的元数据抽象层。
4. **模型关系复杂**：存在单表、扩展表、主从、主从主从等关系，上层组件难以统一理解和使用这些关系。
5. **继承与扩展能力弱**：模型间的强继承（新增、修改、删除字段）缺乏机制，字段冲突无法自动识别和处理。

因此，我们需要一套**基于元数据模型驱动的底层框架**，将模型定义、属性配置、关系管理、变更影响等统一管理起来，为上层应用（表单、表格、过滤器、详情、导出、打印等）提供稳定、可扩展、可配置的能力。


## 2、目标

构建一套**元数据模型驱动开发框架**，实现以下核心目标：

1. **统一模型定义**
    - 支持多种模型关系结构（单表、扩展表、一主多从、主从主从）。
    - 支持多种数据来源范围（单库组合、跨库组合，明确排除支撑/SaaS跨库）。
    - 支持自定义数据来源。

2. **标准化属性集**
    - 每个模型属性包含：字段名（小驼峰/自定义）、类型（varchar→string，bigint→number）、默认值、显示名称。
    - 属性可分组为子集，供不同场景复用。

3. **模型归属与组织**
    - 支持模型分组，一局业务领域进行分组
    - 模型→模块的可选关联，便于治理和查找。

4. **模型变更与依赖管理**
    - 明确上下游依赖关系，支持强耦合（上游强制控制下游引用）和低耦合（下游不受影响）两种策略。
    - 当模型变更（如字段改名）时，能够自动检测并提示下游影响，必要时自动同步。

5. **强继承机制**
    - 支持模型之间的强继承（新增字段、修改字段、删除字段）。
    - 能够检测并处理字段冲突（如子模型修改了父模型的字段定义）。
    - 继承关系可被上层组件理解，用于生成完整模型视图。

6. **为上层应用提供稳定能力**
    - 表单：基于模型自动生成字段输入控件。
    - 表格：基于模型自动生成列、排序、筛选。
    - 过滤器：基于模型属性自动生成查询条件组件。
    - 详情展示页面：自动布局字段展示。
    - Excel 导出、PDF 打印：基于模型及当前数据自动导出/打印。
    - 其他（如 API 生成、校验规则等）可扩展。

## 3、需求

### 3.1 元数据定义与管理

#### 3.1.1 模型定义
- 支持创建、修改、删除模型。
- 模型必须包含：
    - 唯一标识
    - 名称
    - 模型关系结构类型（单表、单表扩展表、一主多从、主从主从）
    - 数据来源范围（单库组合 / 跨库组合）
    - 自定义数据来源描述（若适用）

#### 3.1.2 属性定义
每个属性必须支持：
- 字段名（小驼峰格式，也可自定义命名）
- 字段别名
- 类型（varchar→string，bigint→number，可扩展）
- 默认值
- 显示名称
- 所属属性子集（可选）

#### 3.1.3 属性子集
- 支持将多个属性组合成子集，供上层场景（如某类表单）复用。

### 3.2 模型关系与来源

#### 3.2.1 关系结构
  系统需支持以下四种模型关系结构，用于描述业务实体间的数据存储与关联方式：

| 模式 | 说明 | 典型场景 |
|------|------|----------|
| 单表模式 | 所有属性存储在一张数据库表中 | 简单实体，无复杂关联 |
| 单表扩展表模式 | 主表 + 垂直分表（扩展字段存另一张表） | 主表固定字段，动态/可选字段分开存储 |
| 一主多从 | 一个主表对应多个从表（一对多） | 订单主表 + 订单明细表 |
| 主从主从模式（主子孙结构） | 主-从-孙三层嵌套关系 | 订单 → 产品 → 产品配件 |


| 页面展示名 | 枚举编码 | 原页面名 | 可选用实体结构 | 场景释义 |
| ---- | ---- | ---- | ---- | ---- |
| 列表视图 | LIST_EDIT | 行编辑页面 | 标准单表、横向拓展单表 | 表格直接行内编辑，多用于基础资料维护 |
| 主子明细视图 | MASTER_DETAIL | 主从页面 | 全四种（单表 / 拓展单表 / 一主多子 / 主子孙） | 上方主表单、下方明细表格，适用于订单、入库等单据业务 |
| 树形列表视图 | TREE_TABLE | 树表页面 | 标准单表、横向拓展单表、一主多子 | 左侧分类树，右侧数据列表，按树形维度筛选数据 |
| 树形卡片视图 | TREE_CARD | 树卡页面 | 标准单表、横向拓展单表、一主多子 | 左侧树形导航，右侧卡片式表单，多用于档案查看维护 |

框架需提供关系解析器，供上层组件获取完整模型结构（如嵌套表单、主子表表格）。

#### 3.2.2 数据来源范围
- 明确标识模型是单库组合还是跨库组合。
- **不支持**支撑库、SaaS 库跨库组合，框架需做能力限制或提示。
- 支持自定义数据来源（如接口、视图、动态 SQL），提供扩展点，比如手动添加模型。
- 支持从数据库选择表字段形成模型配置后，手动添加部分列（数据库不存在）

### 3.3 模型归属与模块关联

- 支持模型分组（如“入库管理模型”“计划管理模型组”、“库存管理模型”）。
- 可选将模型关联到一个或多个模块（模块由上层业务定义）。
- 提供按分组/模块查询模型的 API。

### 3.4 模型变更与依赖管理

#### 3.4.1 依赖关系记录
- 框架自动或手动记录模型被哪些上游组件（表单、表格、导出等）引用。
- 支持模型之间的继承依赖（强继承关系）。

#### 3.4.2 耦合策略
- **强耦合**：上游（表单、表格等）字段名称等直接绑定模型字段。当模型字段改名，需强制上游一起变更，框架可提供“同步更新引用”能力。
- **低耦合**：下游组件使用模型字段的别名或快照，模型变更不影响下游，仅提示变更信息。

#### 3.4.3 变更影响分析
- 模型字段新增、修改、删除时，自动分析下游影响范围（哪些组件、哪些导出、哪些继承模型）。
- 提供变更影响报告，支持人工确认或自动同步（在强耦合策略下）。

### 3.5 强继承机制



#### 3.5.1 继承能力


框架允许**上层组件**（如表单、表格、过滤器、详情页、导出配置等）**强继承**一个或多个模型的定义，并在继承基础上进行字段级别的定制，而无需修改原始模型元数据。

每个组件可对继承的模型进行以下操作：

- **新增字段**  
  组件可添加模型原本不存在的字段（如用于 UI 交互的临时字段、计算字段等）。新增字段仅对当前组件生效，不影响模型本身。

- **修改字段**  
  组件可修改继承而来的字段的属性，例如：
    - 更改显示名称
    - 覆盖默认值
    - 修改类型（在兼容范围内，如 varchar → 下拉选项枚举）
    - 调整可见性、必填性、校验规则等  
      修改仅作用于该组件，原模型字段定义保持不变。

- **删除字段**  
  组件可屏蔽模型中的某些字段，使其在该组件中不出现（如表格中隐藏某列、表单中不渲染某输入项）。删除仅对该组件有效。

> **示例**：  
> 订单模型包含字段 `orderNo`, `amount`, `status`。
> - 一个**表单组件**继承订单模型，新增了 `confirmAmount` 字段用于二次校验，并修改 `amount` 字段的显示名称为“订单金额（元）”。
> - 一个**表格组件**继承同一订单模型，删除了 `status` 字段（不在表格中展示），并修改 `orderNo` 的列宽等展示属性。



#### 3.5.2 字段冲突处理


当组件对继承字段的修改与模型原始定义发生冲突时（例如类型不兼容、默认值语义冲突），框架必须检测并处理冲突。

冲突场景示例：
- 组件将模型中的 `amount` 字段（`bigint`→`number`）修改为 `string` 类型，但后续数据绑定、导出等操作可能无法兼容。
- 组件将模型的必填字段改为非必填，导致数据完整性风险。

**处理策略**（可配置，默认采用安全策略）：
1. **报错禁止**：不允许此类冲突修改，组件必须调整其定制内容。
2. **以组件为准**：框架信任组件的定制，但记录警告日志。
3. **以模型为准**：忽略组件的冲突修改，仍使用模型原始定义。

框架应提供冲突检测 API，供开发者在设计组件时提前发现并解决冲突。




#### 3.5.3 继承视图
对于任意一个组件，框架可根据其继承的模型及定制内容，计算出该组件的**完整字段视图**（即模型原始字段 + 组件新增字段 - 组件删除字段，并应用组件修改后的字段属性）。

- 上层应用（如表单渲染器、表格生成器）直接使用该完整视图，无需感知组件内部的继承与定制细节。
- 完整视图可被缓存，并随模型变更或组件定制变更而自动失效刷新。

> **与模型变更的关系**：  
> 当原始模型发生变更（如增加字段、删除字段、修改类型）时，框架会分析该变更对所有继承了该模型的组件的影响，并提示开发者检查组件的定制是否存在新的冲突（见 4.3 变更影响分析）。强继承机制确保了组件可以在模型稳定演进的同时保留其个性化能力。


### 3.6 上层应用支撑能力

#### 3.6.1 表单生成
- 给定模型（或继承后的完整模型），自动生成表单布局（默认按属性顺序）。
- 支持字段级控制（隐藏、只读、必填、校验规则），规则可基于属性元数据配置。

#### 3.6.2 表格生成
- 自动生成列配置（显示名称、字段名、类型）。
- 支持排序、筛选（基于类型推断筛选组件）。
- 支持操作列（编辑、删除等），操作可通过元数据扩展。

#### 3.6.3 过滤器生成
- 基于模型属性自动生成过滤组件（输入框、下拉、日期范围等）。
- 支持组合条件（且/或），可由上层自定义。

#### 3.6.4 详情展示
- 自动生成详情页布局（字段名 + 值）。
- 支持分组展示（利用属性子集）。

#### 3.6.5 Excel 导出 & PDF 打印
- 基于模型定义导出当前数据（支持主子表数据导出）。
- 导出/打印的列可配置（默认全部字段）。
- 支持自定义模板，但仍以模型为数据基础。

#### 3.6.6 扩展机制
- 框架提供 SPI 或插件点，允许上层覆盖默认生成逻辑（如自定义表单控件、自定义列渲染）。

### 3.7 非功能性需求

- **性能**：元数据读取需缓存，避免每次请求解析模型结构。
- **兼容性**：支持增量变更模型，不影响线上运行实例。
- **可观测性**：记录模型变更历史、依赖关系变更。
- **易用性**：提供可视化管理界面（至少 CLI 或 API）管理模型、属性、继承关系。
