## 一、模型结构类型（数据组织方式）

系统需支持以下四种模型关系结构，用于描述业务实体间的数据存储与关联方式：

| 模式 | 说明 | 典型场景 |
|------|------|----------|
| 单表模式 | 所有属性存储在一张数据库表中 | 简单实体，无复杂关联 |
| 单表扩展表模式 | 主表 + 垂直分表（扩展字段存另一张表） | 主表固定字段，动态/可选字段分开存储 |
| 一主多从 | 一个主表对应多个从表（一对多） | 订单主表 + 订单明细表 |
| 主从主从模式（主子孙结构） | 主-从-孙三层嵌套关系 | 订单 → 产品 → 产品配件 |

> **隐含需求**：模型定义需支持配置关系类型、主从关联字段（外键）、级联规则等。

## 二、模型选择范围（数据源与库分布）

| 范围 | 约束条件 |
|------|----------|
| 单库组合 | 所有模型表位于同一数据库（支持不同表但同一库） |
| 跨库组合 | 模型表可跨不同数据库，但**不支持“支撑”和“SaaS跨库”**（需明确排除场景） |
| 自定义数据来源 | 允许用户自定义数据获取逻辑（如SQL、API、视图等）而非仅物理表 |

> **关键点**：跨库需明确“支撑库”和“SaaS库”的定义及禁止原因（性能、事务、安全？）；自定义数据来源需提供配置界面与权限控制。

## 三、模型属性集（字段定义）

每个属性需配置以下元数据：

| 配置项 | 规则与示例 |
|--------|-------------|
| 属性名 | 数据库字段名，**小驼峰格式**（如 `userId`）；或支持自定义命名（可能用于展示或映射） |
| 类型 | `varchar` → 前端/逻辑层映射为 `string`；`bigint` → 映射为 `number`（注意精度） |
| 默认值 | 可设置静态默认值或表达式（需明确） |
| 显示名称 | 用于UI展示的中文/多语言标签 |

> **待澄清**：是否支持其他数据库类型（datetime、decimal、boolean）？类型映射是否可扩展？

## 五、模型变更及上下游依赖关系

模型发生变更（字段增删改、结构变化）时，需控制对下游引用的影响。

### 5.1 影响分析要求
- 系统应能**自动识别下游依赖**（哪些模型、视图、API、表单引用了当前模型）。
- 变更前提示影响范围。

### 5.2 耦合强度控制

| 耦合类型 | 行为 |
|----------|------|
| **强耦合** | 上游模型修改（如字段改名）会**强制同步**到所有下游引用。上游对下游有绝对控制权，下游无法保留旧名。 |
| **低耦合** | 下游不受上游模型变化影响；上游修改后下游可能出现失效引用，需人工适配。 |

> **典型场景**：表单字段名称引用了模型字段。强耦合下修改模型字段名 → 表单字段名自动改；低耦合下表单字段名仍为旧值，导致运行错误。

## 六、强继承机制（模型间继承）

支持一个模型“强继承”另一个模型，类似面向对象的类继承，需处理以下变更操作：

| 操作 | 行为要求 |
|------|----------|
| **新增字段** | 子模型自动获得父模型新增字段；可覆盖默认值或显式配置 |
| **修改字段** | 父模型字段类型、默认值、显示名修改时，子模型同步变更（除非子模型显式覆盖） |
| **删除字段** | 父模型删除字段，子模型同步删除；若子模型依赖该字段，需给出冲突警告 |
| **字段冲突** | 父子模型定义了同名但类型/默认值不同 → 需定义冲突解决策略（如：子模型优先、禁止、抛出异常） |

> **注意**：强继承不同于“模型关系”，它更偏向元数据层面的复用与派生（如基础模型 + 扩展模型）。需明确是否支持多继承、继承层级深度等。
